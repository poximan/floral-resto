ARQUITECTURA DETALLADA DEL ECOSISTEMA FLORAL-RESTO

1. Objetivo general
Disenar un ecosistema compuesto por:
- un server web desplegado en la LAN del comercio, con exposicion publica mediante Cloudflare Tunnel sobre `/carta` y `/gestion`
- una app movil Android de consulta para el encargado, conectada exclusivamente por MQTT

El sistema debe atender tres actores:
- cliente de mesa
- mozo / atendedor
- encargado

El criterio rector es fuente de verdad unica en el backend de dominio. Ningun frontend altera verdad por fuera de contratos explicitos.

2. Vista de alto nivel
La solucion se organiza en microservicios dentro de `./resto_server` y una app Android separada en `./resto_app`.

Componentes principales del lado server:
- `reverse-proxy`: unico nginx frontal, sirve `web-carta` y `web-gestion`, resuelve rutas validas e invalidas y proxifica `/api` al backend
- `gateway`: entrada API HTTP/WebSocket, autenticacion web, exposicion de assets de menu y adaptacion de contratos para frontends
- `domain`: orquestador de negocio, FSM, reglas de sesion, persistencia en PostgreSQL, calculo de dashboard y coordinacion de especialistas internos
- `mqtt-publisher`: adaptador exclusivo entre el dominio y HiveMQ Cloud para la app Android
- `web-carta`: frontend Vue publico para clientes de mesa
- `web-gestion`: frontend Vue interno para mozo y encargado
- `postgres`: persistencia unica del sistema

Componentes externos:
- `HiveMQ Cloud`: broker MQTT administrado externamente
- `Cloudflare Tunnel`: acceso publico a la vista cliente, configurado con URL dinamica

3. Responsabilidades por servicio

3.1 `reverse-proxy`
Responsabilidad dominante:
- ser la unica capa HTTP publica y local para navegadores

Responsabilidades concretas:
- servir `web-carta` en `http://localhost:5175`
- servir `web-gestion` en `http://localhost:5174`
- servir publicamente `/carta` y `/gestion`
- mostrar una sola pantalla de pagina no encontrada para rutas invalidas
- proxificar `/api` y WebSocket hacia `gateway`
- proxificar assets dinamicos de menu hacia `gateway`
- ser el unico target del Cloudflare Tunnel

No debe:
- contener reglas de negocio
- tocar PostgreSQL

3.2 `gateway`
Responsabilidad dominante:
- servir de borde API HTTP/WebSocket para navegadores y proxy frontal

Responsabilidades concretas:
- exponer APIs HTTP para cliente de mesa, mozo y encargado
- gestionar sesiones web autenticadas de mozo y encargado mediante cookie HttpOnly emitida por el gateway
- gestionar apertura de sesion anonima de cliente de mesa
- exponer canales WebSocket por sesion de mesa y por panel interno
- servir assets estaticos de imagenes de menu
- exponer catalogo de imagenes disponibles para el CRUD del encargado
- traducir requests web a comandos contra `domain`
- consultar lectura actual a `domain`
- consumir por HTTP el stream interno de eventos emitido por `domain`

No debe:
- persistir directo en PostgreSQL
- contener reglas de negocio definitivas
- servir HTML de `web-carta` ni `web-gestion`

3.3 `domain`
Responsabilidad dominante:
- ser la fuente de verdad unica de negocio, estado y metricas

Responsabilidades concretas:
- orquestar especialistas internos de persistencia, estado de carrito, presencia de clientes, colas operativas y emision de eventos
- componer servicios internos separados por subdominio para mesa publica, mozo, autenticacion, dashboard, catalogo, mesas y configuracion visual
- encapsular acceso a PostgreSQL mediante una capa dedicada en `domain/src/db`, con DAOs por entidad del modelo, read-model DAOs para consultas analiticas, repositories por caso de negocio y adapters transaccionales por cliente interno
- gestionar mesas habilitadas
- gestionar sesiones de mesa
- gestionar sub-sesiones anonimas de clientes dentro de la mesa
- elegir y reasignar lider de mesa por antiguedad de conexion
- mantener carrito compartido de mesa con autoria actual por `clienteSesionId`
- confirmar cualquier cantidad de pedidos dentro de la misma sesion de mesa
- vaciar el carrito pendiente despues de cada confirmacion para permitir un nuevo pedido
- cerrar mesa, computar cobro y dejar la mesa lista para una sesion nueva
- gestionar consultas, llamados a mozo y pedidos confirmados como eventos formales
- mantener FSM de eventos con estados `pendiente` y `atendido`
- calcular dashboard web y dashboard movil
- administrar CRUD de productos, categorias y mesas
- administrar la configuracion visual de conversion ARS/USD
- detectar concurrencia exclusiva de roles
- publicar eventos de dominio para WebSocket y MQTT a traves de adaptadores consumidores

Es el unico servicio habilitado para leer y escribir PostgreSQL.

3.4 `mqtt-publisher`
Responsabilidad dominante:
- adaptar eventos y pedidos del dominio al protocolo MQTT para Android

Responsabilidades concretas:
- conectarse a HiveMQ Cloud
- publicar snapshots iniciales y novedades del dashboard
- publicar datasets historicos por rango a pedido del movil
- procesar `login_request` del encargado movil
- rechazar un segundo movil activo
- recibir comando `cerrar_sesion_web_encargado` y traducirlo a evento de dominio
- consumir por HTTP el stream interno de eventos emitido por `domain`

No debe:
- recalcular reglas de negocio
- tocar PostgreSQL

3.5 `web-carta`
Responsabilidad dominante:
- experiencia cliente de mesa

Pantallas base:
- ingreso por mesa o formulario de mesa si falta parametro
- carta/menu
- estado del pedido
- pantalla de mesa cerrada con opcion de reingreso

Capacidades clave:
- sesion anonima por navegador/pestania usando `sessionStorage`
- botones `pedir` / `descartar` con contador por producto
- visualizacion de precios en ARS o USD, siendo USD una conversion visual no persistida
- recepcion de cambios del menu por WebSocket
- reflejo en tiempo real del carrito compartido, llamados y consultas

3.6 `web-gestion`
Responsabilidad dominante:
- experiencia de operacion interna para mozo y encargado

Roles y vistas:
- `mozo`: colas, detalle, cierre de conversacion, marcado de recibido, cierre de mesa, ABM simple de mesas, configuracion de conversion ARS/USD
- `encargado`: dashboard, detalle de pendientes/atendidos, CRUD de categorias, CRUD de productos, selector de imagenes disponibles

Autorizacion:
- una sola app Vue con login y rutas protegidas por rol

3.7 `postgres`
Responsabilidad dominante:
- persistencia unica de negocio e historicos

4. Modelo conceptual del dominio

4.1 Mesas
- identificador numerico unico
- habilitada / deshabilitada
- no se puede dar de baja si tiene sesion activa

4.2 Sesion de mesa
Representa una mesa ocupada en un intervalo de tiempo.

Reglas:
- se crea cuando entra el primer cliente a una mesa habilitada sin sesion activa
- permanece abierta hasta que el mozo la cierre explicitamente
- si todos los clientes se desconectan, la sesion no se destruye y queda vacia con su estado operativo vigente
- si luego alguien vuelve a entrar antes del cierre del mozo, se suma a la misma sesion abierta
- puede contener multiples pedidos confirmados
- puede seguir recibiendo consultas y llamados aun despues del pedido confirmado
- se cierra explicitamente por accion del mozo

4.3 Sesion de cliente
Sub-sesion anonima dentro de una sesion de mesa.

Reglas:
- tiene `clienteSesionId` corto de 3 caracteres
- es unico solo dentro de la sesion de mesa
- se conserva al refrescar pagina mediante `sessionStorage`
- puede duplicarse si el navegador duplica pestania, lo cual se acepta
- una desconexion primero entra en gracia parametrizable y no confirma el retiro de inmediato
- la duracion de esa gracia se define por configuracion `.env`
- durante la gracia el cliente sigue contando como conectado
- si reconecta dentro de la gracia, conserva el mismo `clienteSesionId`
- si la salida se confirma y luego vuelve, entra como cliente nuevo con un `clienteSesionId` distinto

4.4 Lider de mesa
Reglas:
- se define por antiguedad de conexion dentro de la sesion activa
- se persiste como relacion 1 a 1 explicita entre sesion y cliente lider
- solo el lider ve el boton `confirmar pedido`
- si el lider se desconecta, hereda el siguiente mas antiguo
- si cualquier cliente se retira de forma confirmada y luego vuelve, entra como cliente nuevo

4.5 Pedido
Reglas:
- el carrito es compartido por toda la mesa
- cada item pendiente del carrito tiene un unico propietario actual por `clienteSesionId`
- solo el propietario actual puede descartar items de su parte del carrito
- si un cliente se retira y existe lider vigente, la propiedad de su parte pendiente del carrito se transfiere al lider actual
- si el cliente que se retira era el lider, el nuevo lider hereda tambien esa parte pendiente al asumir
- si la sesion queda vacia, el carrito pendiente queda huerfano hasta el proximo ingreso
- cuando un nuevo cliente entra a una sesion vacia y queda lider, hereda el carrito pendiente huerfano como propio
- la autoria transferida no conserva historial de duenios anteriores
- los precios en carrito son referencia viva al producto hasta confirmar
- si cambia el precio del producto, el carrito refleja el precio mas nuevo
- al confirmar, se congelan precio e informacion relevante en el pedido
- despues de confirmar, todos los contadores vuelven a cero y se puede volver a armar otro carrito pendiente
- una misma sesion de mesa puede emitir multiples pedidos confirmados antes de cerrar la mesa
- el pedido confirmado no se puede cancelar
- el cobro se considera realizado al cerrar la mesa

4.6 Eventos operativos
Tipos:
- consulta
- pedido confirmado para cocina
- llamado a mozo

FSM:
- `pendiente`
- `atendido`

Reglas particulares:
- consulta: una sola abierta por mesa; se cierra por accion explicita de cliente o mozo; al cerrarse pasa a `atendido`
- llamado a mozo: no puede abrirse otro mientras exista uno pendiente
- pedido confirmado: entra en cola de cocina y el mozo lo marca `recibido`

5. Comportamiento en tiempo real

5.1 WebSocket
Se usara WebSocket para navegadores.

Canales logicos:
- canal de sesion de mesa para cliente
- canal interno para panel de mozo / encargado

Eventos relevantes:
- `mesa_actualizada`
- `pedido_actualizado`
- `consulta_actualizada`
- `llamado_actualizado`
- `cola_actualizada`
- `dashboard_actualizado`
- `menu_actualizado`
- `mesa_cerrada`
- `sesion_expirada`

Reglas:
- reconexion automatica ante corte breve
- si la sesion ya no existe, la UI informa la situacion y vuelve al flujo de ingreso
- cuando el encargado cambia el menu, el cliente recibe `menu_actualizado`, vuelve a pedir catalogo por HTTP y redibuja sin cortar sesion
- si un producto fue desactivado y estaba en carrito no confirmado, se elimina del carrito y se muestra mensaje explicito al cliente

5.2 MQTT
Se usara exclusivamente para Android.

Principios:
- Android no usa HTTP
- el primer snapshot y todas las novedades viajan por MQTT
- los rangos historicos tambien se solicitan y responden via MQTT

Mensajes logicos:
- login request / response
- snapshot inicial
- novedades de dashboard
- novedades de colas
- pedido de dataset historico por rango
- respuesta con dataset historico completo
- comando de cerrar sesion web del encargado

6. Reglas de autenticacion y exclusividad

6.1 Cliente de mesa
- sin login usuario/password
- identificacion por mesa + sesion anonima

6.2 Mozo web
- credenciales configurables por `.env`
- un solo mozo web activo
- la sesion web autenticada viaja en cookie HttpOnly del navegador; el frontend no persiste el token
- si un segundo intenta loguear y las credenciales son validas, la sesion actual se reemplaza de inmediato por el nuevo login
- no existe ventana de inactividad ni chequeo de actividad reciente para conservar la sesion del mozo actual

6.3 Encargado web
- credenciales configurables por `.env`
- un solo encargado web activo
- la sesion web autenticada viaja en cookie HttpOnly del navegador; el frontend no persiste el token
- si un segundo intenta loguear, se evalua actividad de sesion web
- si hubo inactividad por al menos 20 minutos, se permite reemplazo
- la app Android puede forzar el cierre de la sesion web del encargado

6.4 Encargado movil
- un solo dispositivo Android activo en MQTT
- si entra un segundo, se rechaza y se informa la causa
- configuracion sensible cargada desde `local.properties` e inyectada al `BuildConfig` durante el build

7. Persistencia y lineamientos de base de datos

Tablas principales sugeridas:
- `roles_web_sessions`
- `mesas`
- `mesa_sesiones`
- `mesa_clientes`
- `mesa_sesion_lideres`
- `mesa_carrito_items`
- `categorias`
- `productos`
- `pedido_sesiones`
- `pedido_items`
- `pedidos_cocina`
- `consultas_master`
- `consultas_detail`
- `llamados_mozo`
- `eventos_auditoria`
- `configuracion_visual`

Lineamientos:
- montos persistidos en ARS
- timestamps persistidos en UTC
- presentacion horaria segun zona configurada por `.env`
- unicidad real de eventos pendientes por mesa en la base
- carrito pendiente con referencia compuesta hacia cliente de mesa cuando tiene duenio
- productos con baja logica
- categorias no se eliminan si tienen productos activos asociados

8. Catalogo y configuracion operativa

8.1 Categorias
- CRUD exclusivo del encargado
- orden explicito por entero
- borrado bloqueado si existen productos activos
- modal con lista de productos bloqueantes

8.2 Productos
- CRUD exclusivo del encargado
- una y solo una categoria obligatoria
- orden visual: primero por orden de categoria, luego alfabetico por titulo dentro de la categoria
- imagen opcional seleccionada desde catalogo de assets existente
- persistir solo nombre de archivo elegido

8.3 Mesas
- ABM simple disponible para mozo
- solo identificador numerico
- no se puede dar de baja mesa con sesion activa

8.4 Conversion ARS/USD
- configuracion operativa editada desde web interna
- solo afecta visualizacion del cliente
- no persiste como dato historico de pedido o mesa
- no alimenta dashboard

9. Dashboard

9.1 Web encargado
Debe mostrar como minimo:
- cantidad total de pendientes por cola
- cantidad total de atendidos por cola
- tiempos medios, minimos y maximos de atencion
- dinero total de la jornada
- dinero recaudado por mesa en lo que va de la jornada

9.2 Cliente de mesa
Solapa "estado del pedido":
- carrito pendiente actual
- pedidos ya enviados en la sesion actual
- estado de llamado a mozo
- estado de consulta abierta
- cantidad de pedidos confirmados enviados
- total monetario de la mesa

9.3 Android
- dashboard similar al interno pero solo lectura
- paginas horizontales tipo pager
- detalle de colas pendientes y atendidas
- indicador online/offline frente al broker
- cache local con Room
- sincronizacion por rangos via MQTT

11. Contratos entre servicios

11.1 Internos server
- comunicacion por HTTP interna
- solo `domain` toca PostgreSQL
- dentro de `domain`, los servicios de negocio no deben contener SQL embebido ni consumir DAOs directamente; deben operar contra repositories, que componen DAOs por entidad y read-models
- `gateway` y `mqtt-publisher` consumen contratos explicitos de `domain`, incluyendo un stream interno SSE por HTTP

11.2 Hacia web
- HTTP para comandos y lecturas puntuales
- WebSocket para actualizaciones

11.3 Hacia Android
- MQTT para login, snapshots, novedades, historicos y comandos remotos

12. Decisiones tecnicas base
- backend en Node.js + Fastify + JavaScript
- frontend web en Vue 3 + Vite
- app movil en Kotlin + Jetpack Compose + Room
- PostgreSQL como base unica
- HiveMQ Cloud como broker MQTT
- logs estructurados por servicio
- sin fallbacks blandos
- contratos duros y fail-fast ante configuracion incompleta
